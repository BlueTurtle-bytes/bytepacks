# apexpack

Build secure, minimal OCI container images from source using [melange](https://github.com/chainguard-dev/melange) and [apko](https://github.com/chainguard-dev/apko) — no Dockerfiles required.

Language support is driven by **YAML profiles** in the `profiles/` directory. Adding a new language means writing one YAML file. No Go code required.

Project-specific overrides live in an optional **`apexpack.yaml`** in your project root — add packages, change env vars, or override the build command without touching the shared profile.

---

## Quick Start

```bash
# Build the CLI
go build -o bin/apexpack ./cmd/apexpack

# Detect your project language
./bin/apexpack detect .

# Preview what will be built (no tools needed)
./bin/apexpack build . --dry-run

# Build a real OCI image
./bin/apexpack build . --tag ghcr.io/myorg/myapp:v1.0

# Scan for CVEs after building
./bin/apexpack scan

# Check for package updates and patch
./bin/apexpack patch
./bin/apexpack patch --apply
```

## Installation

**Prerequisites:** Go 1.24+

```bash
# Clone and build
git clone https://github.com/apexpack/apexpack
cd apexpack
go build -o bin/apexpack ./cmd/apexpack

# Or install directly into your Go bin (available from any directory)
go install ./cmd/apexpack
```

### Install melange and apko

melange and apko are required for real builds. `--dry-run` works without them.

**macOS (Homebrew):**

```bash
brew install chainguard-dev/tap/melange chainguard-dev/tap/apko
```

**Linux:**

```bash
go install chainguard.dev/melange@latest
go install chainguard.dev/apko@latest
```

## Running on macOS

melange sandboxes builds using Linux kernel namespaces (`bubblewrap`), which macOS does not support. apexpack automatically detects macOS and runs both melange and apko inside Linux Docker containers instead.

**Requirements for real builds on macOS:**
- [Docker Desktop](https://www.docker.com/products/docker-desktop/) installed and running
- melange + apko installed via Homebrew (see above)

```bash
# Make sure Docker Desktop is running, then:
apexpack build . --tag myapp:latest
```

| Command | macOS (no Docker) | macOS (Docker running) | Linux |
|---------|:-----------------:|:---------------------:|:-----:|
| `detect` | ✓ | ✓ | ✓ |
| `profiles` | ✓ | ✓ | ✓ |
| `build --dry-run` | ✓ | ✓ | ✓ |
| `build` (real) | ✗ | ✓ | ✓ |
| `scan` | ✓ (needs grype) | ✓ | ✓ |
| `patch` | ✓ | ✓ | ✓ |

### Signing keys

melange signs every `.apk` it builds. apexpack generates a key pair automatically on the first real build:

```
.apexpack-output/
├── melange.rsa        ← private signing key  (add to .gitignore)
├── melange.rsa.pub    ← public key (added to apko keyring automatically)
├── melange.yaml       ← generated build config
├── apko.yaml          ← generated image config
├── packages/          ← built .apk files
├── myapp.tar          ← final OCI image tarball
└── sbom-x86_64.spdx.json  ← SBOM (generated by apko automatically)
```

## Commands

### `apexpack detect [source-dir]`

Scans a source directory against all profiles, sorted by confidence. Shows the detected runtime and framework.

```bash
apexpack detect .
apexpack detect /path/to/my-project
apexpack detect . --profiles-dir /custom/profiles
```

```
Detected 1 match(es) in .:

→ node          90%  framework: nextjs   (matched: [package.json])

To build: apexpack build .
```

---

### `apexpack build [source-dir]`

Detects the language (or uses `--runtime`), merges the language profile with any `apexpack.yaml` in the source directory, generates `melange.yaml` and `apko.yaml`, then runs melange and apko.

```bash
apexpack build .
apexpack build . --tag ghcr.io/myorg/myapp:v1.0.0
apexpack build . --runtime java
apexpack build . --dry-run
apexpack build . --project-name myservice
apexpack build . --output /tmp/my-build
```

| Flag | Default | Description |
|------|---------|-------------|
| `--tag` / `-t` | `<project>:latest` | OCI image reference |
| `--runtime` | auto-detect | Skip detection, use this profile directly |
| `--project-name` | source directory name | Override the APK package name and binary name |
| `--version` | `0.0.1` | Version embedded in the APK package |
| `--output` / `-o` | `.apexpack-output/` | Where configs and image tarball are written |
| `--dry-run` | `false` | Print generated configs, do not run tools |
| `--profiles-dir` | `profiles/` | Directory containing language profiles |

---

### `apexpack scan [output-dir]`

Scans the SBOM produced by the last build for CVEs using grype. Install grype: `brew install grype`.

```bash
# Scan the last build
apexpack scan

# Scan a specific output directory
apexpack scan /path/to/.apexpack-output

# Fail with exit code 1 if any HIGH CVEs found (for CI)
apexpack scan --fail-on high

# SARIF output for GitHub Code Scanning
apexpack scan --format sarif --output results.sarif
```

| Flag | Default | Description |
|------|---------|-------------|
| `--sbom` | `<output-dir>/sbom-x86_64.spdx.json` | Path to a specific SBOM file |
| `--fail-on` | (none) | Exit 1 if CVEs found at this severity: `critical`, `high`, `medium`, `low` |
| `--format` | `table` | Output format: `table`, `json`, `sarif`, `cyclonedx` |
| `--output` / `-o` | stdout | Write scan report to this directory |

---

### `apexpack patch [output-dir]`

Compares installed package versions (from the SBOM) against the latest in the Wolfi package index. With `--apply`, updates `profiles/*.yaml` to pin patched versions.

```bash
# Show what packages have updates available
apexpack patch

# Show updates for a specific output directory
apexpack patch /path/to/.apexpack-output

# Update profile YAML files with pinned patched versions
apexpack patch --apply

# Then rebuild to apply the patches
apexpack build .
```

| Flag | Default | Description |
|------|---------|-------------|
| `--sbom` | `<output-dir>/sbom-x86_64.spdx.json` | SBOM to compare against |
| `--apply` | `false` | Update profile YAML files with pinned patched versions |
| `--profiles-dir` | `profiles/` | Directory containing language profiles |
| `--arch` | `x86_64` | Architecture to check against Wolfi |

### `apexpack profiles`

Lists all loaded language profiles.

```bash
apexpack profiles
```

## Per-Project Configuration: `apexpack.yaml`

Place an `apexpack.yaml` file in your project root to customise the build for that specific project. This file is **optional** — if absent, apexpack uses the language profile defaults unchanged.

### When to use it

The language profile (e.g. `profiles/golang.yaml`) provides defaults that work for most projects. Use `apexpack.yaml` when your project needs something different:

- A C extension that requires a native library at runtime (e.g. SQLite, PostgreSQL client)
- A custom build command (e.g. monorepo with a specific service path)
- Runtime environment variables baked into the image
- A different entrypoint path

### Format

```yaml
# apexpack.yaml — place in your project root alongside go.mod / pom.xml / package.json

runtime: golang   # optional — overrides auto-detection if set

image:
  packages:        # appended to the profile's image.packages list
    - sqlite-libs
    - libpq
  entrypoint: /usr/bin/myservice   # overrides the profile entrypoint if set
  env:             # merged with the profile's image.env (project values win)
    APP_ENV: production
    DB_PATH: /data/app.db

build:
  dependencies:    # appended to the profile's build.dependencies list
    - sqlite-dev
    - postgresql-dev
  env:             # merged with the profile's build.env (project values win)
    CGO_ENABLED: "1"
  command: |       # fully replaces the profile's build.command if set
    mkdir -p ${{targets.destdir}}/usr/bin
    go build -tags sqlite -o ${{targets.destdir}}/usr/bin/myservice ./cmd/myservice
```

### Merge rules

| Field | Behaviour |
|-------|-----------|
| `runtime` | Overrides auto-detection |
| `image.packages` | **Appended** to profile packages — both lists are used |
| `image.entrypoint` | Replaces profile entrypoint |
| `image.env` | **Merged** — project values win on duplicate keys |
| `build.dependencies` | **Appended** to profile build deps |
| `build.env` | **Merged** — project values win on duplicate keys |
| `build.command` | Replaces profile command entirely |

### Example: Go service with SQLite (CGO)

```yaml
# apexpack.yaml
build:
  dependencies:
    - sqlite-dev     # needed at compile time for CGO
  env:
    CGO_ENABLED: "1"   # override the profile's CGO_ENABLED=0

image:
  packages:
    - sqlite-libs    # needed at runtime
```

### Example: Python service with PostgreSQL

```yaml
# apexpack.yaml
build:
  dependencies:
    - postgresql-dev   # psycopg2 needs pg_config at compile time
    - build-base

image:
  packages:
    - libpq            # psycopg2 needs libpq.so at runtime
  env:
    DATABASE_URL: "postgresql://localhost/app"
```

### Example: Monorepo — build a specific service

```yaml
# apexpack.yaml
build:
  command: |
    mkdir -p ${{targets.destdir}}/usr/bin
    go build -o ${{targets.destdir}}/usr/bin/api ./services/api

image:
  entrypoint: /usr/bin/api
```

## How It Works

### The Build Pipeline

```
Source Code
    │
    ▼
detect               Read profiles/*.yaml, match files against source dir
    │                Score by confidence, identify framework from content rules
    ▼
Language Profile     profiles/golang.yaml, profiles/java.yaml, etc.
    │                Defines: detect rules, build command, image packages
    ▼
apexpack.yaml        Optional per-project overrides (packages, env, command)
(project root)       Merged on top of the profile — project values win
    │
    ▼
melange.yaml  ──┐
                │  Generated from the merged profile + project name/version
apko.yaml     ──┘
    │
    ▼
melange              Compiles source → .apk package (Wolfi APK format)
    │                Sandboxed build, reproducible output, signed package
    ▼
apko                 Assembles .apk packages → OCI image + SBOM
    │                Minimal Wolfi base, non-root by default (UID 65532)
    ▼
OCI Image (.tar)     Ready to push to any registry
    │
    ▼
grype (scan)         Scans SBOM for CVEs — can gate the pipeline
```

### Why melange + apko instead of Dockerfile?

| Concern | Dockerfile | melange + apko |
|---------|-----------|----------------|
| Reproducibility | No — `apt-get` results vary | Yes — byte-identical rebuilds |
| SBOM | Manual | Automatic, per-package |
| CVE patching | Rebuild whole image | Swap one APK package |
| Image size | Varies | Minimal — only declared packages |
| Non-root | Manual setup | Default (UID 65532) |
| Build isolation | Shared Docker daemon | Sandboxed per build |

## Language Profiles

### Bundled Profiles

| Profile | Detects | Framework Detection | Build Tool |
|---------|---------|---------------------|------------|
| `golang` | `go.mod` | gin, echo, fiber, grpc, connect | `go build` |
| `java` | `pom.xml`, `build.gradle` | spring-boot, quarkus, micronaut | `mvn package` |
| `dotnet` | `*.csproj`, `*.sln` | aspnetcore, masstransit, orleans | `dotnet publish` |
| `node` | `package.json` | nextjs, nestjs, express, fastify, hono, remix | `npm ci && npm run build` |
| `python` | `requirements.txt`, `pyproject.toml`, `Pipfile` | fastapi, django, flask, aiohttp | `pip install` |
| `webserver` | `index.html`, `vite.config.*`, `angular.json` | angular, vite, react, vue, svelte, hugo | `npm run build` or copy |

### Profile File Format

```yaml
runtime: golang           # unique identifier — matches the --runtime flag
version: "1"
description: "Go application"

detect:
  files:                  # exact filenames — ANY match triggers this profile
    - go.mod
  patterns:               # glob patterns — for variable filenames (e.g. *.csproj)
    - "*.csproj"
  content:                # read a file and check for a string
    - file: go.sum
      contains: "gin-gonic"
      boost-confidence: 0.02   # added to base confidence when matched
      framework: gin            # sets the detected framework name
  confidence: 0.85              # base score (0.0–1.0)

build:
  dependencies:           # APK packages in the BUILD environment only (not in image)
    - busybox
    - go
    - build-base
    - git
  command: |              # shell script; output goes to ${{targets.destdir}}/usr/bin/
    mkdir -p ${{targets.destdir}}/usr/bin
    go build -o ${{targets.destdir}}/usr/bin/app .
  env:
    CGO_ENABLED: "0"
    GOFLAGS: "-trimpath"

image:
  packages:               # APK packages in the FINAL image — keep minimal
    - ca-certificates-bundle
  entrypoint: /usr/bin/app
  run-as: 65532           # never 0 (root)
  ports:
    - "8080"
  env:
    APP_ENV: production
```

### Adding a New Language Profile

1. Create `profiles/rust.yaml`
2. Fill in `runtime`, `detect`, `build`, `image`
3. Verify it loads and detects correctly

```bash
apexpack profiles          # should show rust in the list
apexpack detect /path/to/rust-project
apexpack build /path/to/rust-project --dry-run
```

## CVE Patching

Wolfi packages are patched within hours of a CVE disclosure. The patch workflow:

```bash
# 1. Build the image (produces SBOM automatically)
apexpack build . --tag myapp:v1.0

# 2. Scan for CVEs
apexpack scan
#    PACKAGE                  INSTALLED      LATEST         CVEs
#    ca-certificates-bundle   20241121-r0    20260413-r0    CVE-2024-XXXX

# 3. Check for available patches
apexpack patch
#    ↑ ca-certificates-bundle  20241121-r0 → 20260413-r0

# 4. Pin the patched versions in all affected profiles
apexpack patch --apply
#    golang.yaml:
#      ↑ ca-certificates-bundle → ca-certificates-bundle=20260413-r0

# 5. Rebuild — new image uses patched packages
apexpack build . --tag myapp:v1.1
```

For project-specific pinning, use `apexpack.yaml`:

```yaml
# apexpack.yaml — pin to a known-good version after patching
image:
  packages:
    - ca-certificates-bundle=20260413-r0
```

## Testing Locally

### Step 1 — Build the binary

```bash
cd apexpack
go build -o bin/apexpack ./cmd/apexpack
./bin/apexpack --help
```

### Step 2 — Run the unit tests

```bash
go test ./...
```

### Step 3 — Test detection

```bash
# apexpack detects itself as a Go project
./bin/apexpack detect .

# List all loaded profiles
./bin/apexpack profiles

# Test against nimbopacks (another Go project)
./bin/apexpack detect ../nimbopacks
```

### Step 4 — Test the build pipeline (dry-run, no tools needed)

```bash
# See exactly what melange.yaml and apko.yaml would be generated
./bin/apexpack build . --dry-run

# Test a specific runtime
./bin/apexpack build . --dry-run --runtime java
```

### Step 5 — Test the `apexpack.yaml` override

```bash
# Create a test project
mkdir /tmp/test-go && cd /tmp/test-go
go mod init example.com/testapp
cat > main.go << 'EOF'
package main
import "fmt"
func main() { fmt.Println("hello") }
EOF

# No apexpack.yaml — uses profile defaults
cd /path/to/apexpack
./bin/apexpack build /tmp/test-go --dry-run

# Add an apexpack.yaml override
cat > /tmp/test-go/apexpack.yaml << 'EOF'
image:
  packages:
    - sqlite-libs
  env:
    APP_ENV: production
build:
  env:
    CGO_ENABLED: "1"
EOF

# Dry-run again — should show sqlite-libs and CGO_ENABLED=1
./bin/apexpack build /tmp/test-go --dry-run
# Look for:
#   → Merged apexpack.yaml project overrides
#   CGO_ENABLED: "1"
#   - sqlite-libs
```

### Step 6 — Real build (requires Docker Desktop on Mac)

```bash
# Build nimbopacks as a real OCI image
./bin/apexpack build ../nimbopacks --tag nimbo:latest

# Load and run it
docker load < ../nimbopacks/.apexpack-output/nimbopacks.tar
docker run --rm --platform linux/amd64 nimbo:latest-amd64 --help
```

### Step 7 — Scan and patch

```bash
# Install grype (Mac)
brew install grype

# Scan the last built image
./bin/apexpack scan ../nimbopacks/.apexpack-output

# Check for package updates
./bin/apexpack patch ../nimbopacks/.apexpack-output
```

## Project Structure

```
apexpack/
│
├── cmd/
│   └── apexpack/
│       └── main.go         CLI — detect, build, scan, patch, profiles, version
│
├── internal/
│   ├── types/
│   │   └── types.go        ALL data structures — Profile, ProjectConfig, BuildPlan, etc.
│   │
│   ├── profile/
│   │   └── profile.go      LoadAll(), Load(), LoadProjectConfig(), MergeProjectConfig()
│   │
│   ├── detect/
│   │   └── detect.go       Run(), Best() — matches profiles against a source directory
│   │
│   ├── build/
│   │   └── build.go        Plan(), Run(), MarshalMelange(), MarshalApko()
│   │
│   └── patch/
│       └── patch.go        Check(), ApplyToProfile() — CVE and package update workflow
│
└── profiles/
    ├── golang.yaml
    ├── java.yaml
    ├── dotnet.yaml
    ├── node.yaml
    ├── python.yaml
    └── webserver.yaml
```

### Data Flow

```
profiles/*.yaml
      │  profile.LoadAll()
      ▼
[]*types.Profile
      │
      │  detect.Best(profiles, srcDir)
      ▼
*types.DetectResult  { Profile, Confidence, Framework }
      │
      │  profile.LoadProjectConfig(srcDir)   ← reads apexpack.yaml if present
      │  profile.MergeProjectConfig()        ← project overrides applied here
      ▼
*types.Profile  (merged: profile defaults + project overrides)
      │
      │  build.Plan(profile, opts)
      ▼
*types.BuildPlan  { Melange MelangeConfig, Apko ApkoConfig }
      │
      │  build.Run(plan, opts)
      ▼
melange → packages/x86_64/*.apk
apko    → myapp.tar + sbom-x86_64.spdx.json
```

### Key Design Decisions

**1. `types.go` is the centre** — every package imports it. Nothing imports from `cmd/`. Circular imports are impossible.

**2. Detection is read-only** — `detect.Run()` only reads files. No writes, no network, no side effects.

**3. Plan and Run are separated** — `build.Plan()` generates structs. `build.Run()` writes YAML and runs tools. `--dry-run` uses Plan only.

**4. Profiles over code** — language knowledge lives in `profiles/*.yaml`, not Go code. Adding a language = one YAML file. No Go required.

**5. `apexpack.yaml` for exceptions** — the profile covers 95% of projects. `apexpack.yaml` covers the 5% that needs something different. No dependency auto-detection — if a project needs an extra package, the developer declares it explicitly.

**6. Structs → `yaml.Marshal`** — melange and apko configs are Go structs serialised with `yaml.v3`, not string templates. The marshaller handles indentation and quoting — no formatting bugs.
